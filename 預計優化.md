# Internal AI Copilot Go Backend 預計優化

## 目的
這份文件先記錄目前文件流程的體感、問題點、以及預計優化方向。

這不是當前需求的 implementation 文件，而是後續整理開發流程時的參考基線。

原則：
- 先記錄清楚
- 這次需求先完成
- 文件流程優化等本輪需求收尾後再做

## 目前體感
目前這套分法整體是可用的，而且對這個專案有幫助。

原因：
- Go backend 不是單點功能，而是跨 `grpcapi / gatekeeper / builder / infra / aiclient / output`
- 需求常常會牽涉 contract、驗證責任、prompt assembly、資料模型
- 如果沒有先對齊文件，後續 coding 很容易把邊界混掉

目前的優點：
- `PLAN` 可以承接跨模組設計與大方向
- module `spec` 可以承接責任與邊界
- module `BDD` 可以承接 scenario 與 acceptance
- `CODE_REVIEW` 可以承接 current code 狀態與風險

目前的問題不是「要不要寫文件」，而是：
- code 改完後，文件如果沒有立即 sync，很容易半拍落後
- `planned` 與 `current` 有時會混在同一份文件裡
- 同一件事可能同時在 `PLAN / spec / BDD` 寫太滿，造成維護負擔

## 總結判斷
目前分法基本上是對的，不需要推翻。

真正要優化的是：
- 每份文件的唯一職責
- planned vs current 的標示
- code 後的 sync 規則
- 哪些內容應該從 BDD 移回 spec / PLAN

## 建議的文件職責

### 1. `PLAN.md`
只放：
- 跨模組設計
- 大型流程
- 為什麼這樣切
- planned design
- migration direction
- collection / contract 的高階模型

不應放太多：
- module 細節驗證規則
- code-backed scenario 級 acceptance
- 目前 implementation 的細節清單

### 2. `DEVELOPMENT.md`
只放：
- 開發流程規範
- BDD-first / TDD-first 規則
- 文件同步規則
- 測試與驗證要求
- collaboration / review 規範

不應放：
- 某個 module 的實際業務規則
- current code 狀態說明

### 3. module `*_spec.md`
只放：
- 模組責任
- 模組邊界
- 資料流
- 固定規則
- 欄位語意
- runtime 限制

適合放在 spec 的內容：
- `builder` 的 strategy / factory 責任
- `gatekeeper` 的 validation 邊界
- `infra` 的資料欄位與 ownership

不應放太多：
- 長篇 planned roadmap
- code review 結論
- 太細的 Given/When/Then

### 4. module `*_BDD.md`
只放：
- code-backed scenario
- acceptance criteria
- Given / When / Then
- 測試保護到哪裡

BDD 應偏向：
```text
Given ...
When ...
Then ...
```

不應放太多：
- planned design
- 大篇幅 schema 討論
- migration note
- 架構理由

### 5. `CODE_REVIEW.md`
只放：
- current implementation 狀態
- 行為與 code 對照
- 風險
- 限制
- 測試覆蓋與驗證方式

不應放：
- planned design
- 未落地的 schema 提案
- 未實作的理想行為

## 建議的內容搬移方向
目前最值得優化的是 BDD 內偏設計稿的內容。

### 應從 BDD 移回 `PLAN` 的內容
- planned design
- 下一版方向
- migration 說明
- 大型架構理由

### 應從 BDD 移回 `spec` 的內容
- 欄位語意
- repository / collection 規則
- source / rag ownership
- prompt assembly 固定規則

### BDD 應保留的內容
- 已落地 scenario
- 驗收條件
- 錯誤情境
- 目前測試保護範圍

## Planned vs Current 標示規則
之後所有文件建議統一這套：

```text
current
  = 已落地 code / test / behavior

planned
  = 已討論、已對齊、但尚未落地
```

規則：
- `PLAN` 可以同時寫 `current` 與 `planned`
- `spec` 以 `current` 為主，必要時才補小段 `planned`
- `BDD` 原則上只寫 `current`
- 若 `BDD` 必須提 planned，必須放在獨立區塊，且清楚標示不是 code-backed

## 文件同步建議流程

```text
需求討論
   │
   ▼
先改 PLAN / spec / BDD
   │
   ▼
再寫 code
   │
   ▼
跑驗證
   │
   ▼
最後對照 code 更新 CODE_REVIEW
```

更細一點：

```text
1. 行為改了
   -> 先改 PLAN / spec / BDD

2. code 落地
   -> 跑測試 / compile / emulator 驗證

3. 驗證過了
   -> 再開 CODE_REVIEW
   -> 必須重新讀 code 後再寫
   -> 不可憑記憶同步
```

## 建議的同步檢查表
每次較大變更後，至少檢查：

- `PLAN` 是否還在寫舊方向
- `spec` 是否還保留已失效欄位
- `BDD` 是否把 planned 當成 current
- `CODE_REVIEW` 是否仍在描述舊 code
- 錯誤碼、欄位名、流程圖是否和 production code 一致

## 是否需要切更多 module 文件
目前判斷：
- 核心 module 切分是合理的
- 不建議退回成「全專案只靠一份大文件」
- 也不建議每個小東西都拆一套

目前比較合理的核心模組文件範圍：
- `builder`
- `gatekeeper`
- `grpcapi`
- `infra`

其他模組如果很薄，維持較輕量即可。

判斷原則：

```text
這個模組是否：
  1. 有明確邊界？
  2. 常被反覆討論？
  3. 容易和別模組責任混淆？
  4. 有獨立驗證規則？

若多數答案是 yes
  -> 值得有自己的 spec / BDD

若多數答案是 no
  -> 不需要重文件化
```

## 這次流程的經驗結論
這次實際操作下來，最有價值的不是「文件很多」，而是：
- 先對齊
- 再 coding
- coding 後再回頭 sync

最容易出問題的地方是：
- code 已經改了，但 `spec / BDD / PLAN` 還留在上一版
- `CODE_REVIEW` 被當成設計文件，而不是 current implementation 摘要
- BDD 吃進太多 planned design

## 後續建議優化順序

### Phase 1：先定義文件唯一職責
- 把 `PLAN / spec / BDD / CODE_REVIEW` 的責任寫死
- 後面所有 sync 都照這套走

### Phase 2：收斂 BDD 內容
- 把偏設計稿的段落移回 `PLAN / spec`
- 讓 BDD 更像 acceptance 文件

### Phase 3：補同步規則
- 每次 code merge 前檢查：
  - `PLAN`
  - changed module `spec`
  - changed module `BDD`
  - `CODE_REVIEW`

### Phase 4：視情況補模板
- 如果後面變更多，可再補：
  - 新 feature 文件模板
  - code review sync checklist
  - module spec / BDD 的固定骨架

## 暫時結論
目前流程不需要推翻。

目前最合理的策略是：
- 保留現在的文件分法
- 之後再做小幅整理
- 優先收斂文件職責，而不是增加更多文件

一句話：

```text
目前流程可用；
下一步不是重做，而是把每份文件各自負責什麼收得更乾淨。
```
